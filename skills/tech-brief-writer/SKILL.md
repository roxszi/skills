# 技术简报撰写 skill（tech-brief-writer）

### 何时用

用户要求为某项**技术/方法/仪器原理**（重点：分析化学、物理化学、光谱学、化学计量学交叉域）撰写可长期存档的技术简报 / 备忘。典型触发：
- "帮我写一个 XX 技术的技术简报，我存下来备忘"
- "看一下这篇文章用的技术，整理成档案"
- "把 XX 方法的原理 + 公式 + 应用整理一份 md"

> ⚠️ **涉及基础依赖技术时的特别提示**：撰写前先判断是否属于"主体 + 依赖"复合技术（如 EC-SERS 主体 + 拉曼基础依赖）；若是，参见 §三·3.4 的技术依赖处理规则，避免教学法漂移（pedagogical drift）。

### 目录规范

每个技术**独立一个文件夹**，放在归档工作目录下：

```
<tech-name>-technical-brief/
├── <tech>_技术简报.md
├── <tech>_演示.html                       # 可选：交互演示
├── assets/                                # 可选：所有外部 SVG / 各类资源
│   ├── xx_schematic.svg
│   ├── xx_spectrum.svg
│   └── ...
├── scripts/                               # 可选：可执行 / 可复现的脚本（首推 .html）
│   ├── generate_xx.html                   # 浏览器内可直接打开：脚本即代码、结果即页面
│   ├── generate_xx.mjs                    # 可选：Node 端运行（需 bun/node 环境）
│   └── ...
└── README.md                              # 可选：目录索引（仅当文件数量较多 / 多简报共建 / 对外分享时）
```

> 📁 **scripts/ 首选 `.html` 的理由**：
> - 浏览器 = 运行时 + IDE + 查看器，**三者合一**；
> - 所有设备（PC / 平板 / 手机）都有浏览器，零环境依赖；
> - 文件即程序：`<script>` 里的代码就是脚本本体，保存即生效（刷新即看结果）；
> - HTML 在这里**兼有"脚本"和"产物"双重身份**——既是 `generate_xx.html`，也可以直接拷贝关键 SVG 到 `assets/` 落档。

## 一、MD 简报章节模板

按需裁剪，但**加粗项 / ★ 项为硬性要求**。

> 📌 **章节模板 vs 配套细化规范**：本章是"每节里要写什么"的硬性要求骨架；具体写作格式（公式、参考文献、SVG、依赖处理）单列为第三大节的细化规范，避免与本章互相重复。改规范只动对应章节。
>
> 📌 **依赖地图必填**：本模板第 13 章（依赖地图）是有跨章节生效的依赖类技术的硬性产物，详见 §三·3.4。
> 
> 📌 **formatter 元数据必填**：markdown 顶部的 yaml formatter 元数据（文档性质 / 最后修订日期 / 作者 / 发起人 / 其它有必要写在文档头的内容）。
> 
> 📌 **目录必填**：位置在 markdown 标题之下，§1 之上。

1. **一句话定义**：blockquote 引用块开篇（中文定义 + 英文全称/缩写）
2. **核心技术原理**
   - 问题背景（可用"传统方案 vs 局限"对比表引入痛点）
   - ★ **一个统一核心洞察（single conceptual anchor）**：把整项技术的精髓凝练成**一张表或一句话**（例：SERDS 的"坐标对偶性"——谁不动谁被差分消掉；LTRS 的"梯度力 vs 散射力平衡"）。这是整篇的灵魂锚点，读者记住这一个就抓住了本质。
3. **原理示意图**：**§3.1 Mermaid（流程 / 因果图）** + **§3.2 外置 SVG（几何 / 谱图）**；二者互补、**禁止用 ASCII art 替代几何 / 谱图**
   - **§3.1 Mermaid**：算法步骤 / 数据流 / 因果链。Mermaid 语法踩坑清单见 §三·3.5
   - **§3.2 外置 SVG**：频谱位置、χ⁽³⁾ 谱形、信号路径、光路几何等几何 / 谱图类内容
     - 一律外置到 `assets/` 下，md 用标准图片语法引用：
       ```markdown
       ![光阱结构示意图](assets/optical_trap_geometries.svg)
       ```
     - **不内嵌** `<svg>` 块到 md 里（部分 markdown 渲染器对 inline SVG 支持不一致）
     - 完整规范（命名 / 字体 / 锚点 / 颜色 / 生成脚本）见 §三·3.3
   - **★ 禁止用 ASCII art（```代码块围栏的字符图）替代几何 / 谱图**：等宽字体宽度不一导致错位、emoji 与中文混排错位、不可缩放、无颜色编码
4. **基本公式方程**
   - 每个关键公式后尽量给 **带入数值的算例（worked example）**（如 "λ=785 nm, Δλ=0.5 nm → Δν̃≈8.1 cm⁻¹"）
   - ★ **适用区间/有效性警示**：注明公式成立的前提与边界（如瑞利区 vs Mie 区、Kasha 规则例外）。**不给未经数学/物理验证的式子，边界必须写清**
   - 公式四要素（公式本体 / 符号说明 / 数值例子 / 适用条件）的统一写法见 §三·3.1。
5. 技术变体 / 发展（可选）：该技术存在 ≥1 个主流变体时必填；否则省略。同样配图 + 公式 + 对比表。
6. **数据处理算法**（当涉及化学计量学时**必做**）：算法 / 原理 / 优势 / 适用场景 四列表。
7. **典型示例应用**
   - 论文信息：标题、期刊年份、**链接 + DOI**
   - 仪器配置表、样品集、处理策略、关键发现
   - ★ **明确标注"用到本技术/变体的具体位置"**（便于核对"本技术是否真在论文中被使用"）
8. **注意事项与工程要点**：⚠️ **踩坑框** + 表格（"要点 / 原因"两列）
9. **优缺点**（优点 / 缺点约束，分列）
10. **参考文献**（末尾必填）：每篇文献必须包含作者 / 标题 / 期刊 / 年份 / 卷期页 / DOI；DOI 必须核实，未核实的一律标注 `（待核实）`；分类排列：经典（按时间正序）→ 综述 → 应用论文；用户提供的应用论文排应用类第一条。**完整规范见 §三·3.2。**
11. 记忆口诀（可选但推荐）：当存在多个核心公式 / 概念需要记住时，一句话助记。
12. **溯源脚注（provenance footer）**：斜体注明"基于谁在何时的什么要求或什么样的讨论进行整理/修订"；**若过程中纠正过错误认知，明确记录下来**（诚实透明，也便于日后追溯）
13. **依赖地图（Dependency Map，末尾必填）**：以一张四列表（依赖强度 / 技术 / 在 A 中的角色 / 是否需要独立简报）声明本简报与外部基础技术的关系。格式与判定准则见 §三·3.4。

> 💡 **填空骨架（template）**：本 skill 同目录下的 [`template.md`](template.md) 与上述 13 章节列表 **1-to-1 对齐**，以 `{占位符}` 形式给出可直接套用的空白简报结构。
> 
> **推荐工作流**：
> 1. 复制 `template.md` → 重命名为 `<技术名>_技术简报.md`
> 2. 按 §一 顺序，逐章替换 `{占位符}` 内容
> 3. 按需裁剪可选章节（§一·2 目录 / §一·7 技术变体 / §一·13 记忆口诀）
> 4. 完成后走 §五 自检清单

## 二、演示（交互式 HTML）规格

### 触发条件

当技术的核心原理涉及**"参数变化 → 光谱/信号形态变化"**等因果关系时（例如：波长差影响差分峰幅度、光阱结构影响探测区域），且能用几个滑块让核心物理量变化趋势可视化时，**必须**生成一份独立 HTML 交互演示。

### 技术要求

- **单文件自包含**：HTML + JS + `<canvas>`，**离线双击可开**
- **技术栈**：JS/TS 优先，ESM 风格；若涉及张量 / ML 再考虑 tfjs（浏览器端）
- **中文页面**：标题、轴标签、图例全部中文
- **高 DPI 适配**：读 `window.devicePixelRatio`，canvas 物理像素 = CSS 尺寸 × dpr
  - ⚠️ **ImageData 索引必须用整数（imgData.width / imgData.height），不能用浮点 plotW / plotH**。否则会产生 小数索引，被自动 `Math.floor` 后落到错误像素的 A 通道，导致热图/位图大面积失色。
  - `canvas.width/height/style.width/height` 任何时候都不在重绘函数里赋值。违反 = 物理像素累积爆炸。
- **参数滑块实时重算重绘**：
  - 简单交互（默认）：`input` 事件 → 重新 `computeData()` → `drawAll()`，vanilla JS 即可，**零依赖**
  - 较复杂交互响应（多 view、组件化、跨滑块共享状态）：**VUE3 渐进式引入** —— 以 CDN 单文件形式挂载，**不要**走 `npm + Vite/webpack`：
    ```html
    <script src="https://unpkg.com/vue@3/dist/vue.global.js"></script>
    ```
    ⚠️ CDN 引入意味着**首次加载需联网**（`vue.global.js` 浏览器缓存后即可离线）；因此务必**在 HTML 中显式声明 "需要联网"**，让用户知道。
- **围绕"核心洞察"设计对比视图**：配色与内容含义匹配（正确用暖绿/错用暖红等），一眼看懂
- **中文注释**：物理参数、函数、算法流程均中文注释
- **物理要忠实**：演示里的公式实现要和 MD 里的公式一致，不能为了好看而失真
- 自绘坐标轴/网格/图例/零线；峰型用洛伦兹（拉曼峰）/高斯（荧光/带）等符合物理的函数
- 若演示中也用到同样的几何/谱图示意（**非交互态**），直接 **`assets/` 单一来源复用** SVG（详见 §三·3.3 第 4 条），避免在演示里用 canvas 重画一遍静态图。
- **交付前自测**（硬性要求）：演示 HTML 完成后，**必须**在本地浏览器打开、逐个交互 ≥ 1 次、运行 ≥ 30 秒，目视无 bug 后才能提交。常见 bug 清单：
  - 选参数时画面**累积变大/变小**（高 DPI 像素累积陷阱）
  - 滑块拖动时画面**不响应**（事件未绑定 / state 未更新）
  - 循环动画（CV 扫描）**越来越快**（`requestAnimationFrame` 累积未取消）
  - 切换下拉时**canvas 残留**（清屏时未用 dpr 重置坐标系）

### 高 DPI Canvas 模板

```js
// 初始化（页面加载时一次性执行）
function initCanvas(canvas) {
  const dpr = window.devicePixelRatio || 1;
  const cssW = canvas.clientWidth || canvas.width;  // CSS 像素
  const cssH = canvas.clientHeight || canvas.height;
  canvas.width  = cssW * dpr;   // 物理像素：只设这一次
  canvas.height = cssH * dpr;
  canvas.style.width  = cssW + "px";
  canvas.style.height = cssH + "px";
  canvas._ctx = canvas.getContext("2d");
  canvas._dpr = dpr;
  canvas._cssW = cssW; canvas._cssH = cssH;
}
// 重绘（每次更新调用）
function redraw(canvas) {
  const { _ctx: ctx, _dpr: dpr, _cssW: w, _cssH: h } = canvas;
  ctx.setTransform(dpr, 0, 0, dpr, 0, 0);   // 只重置坐标系
  ctx.clearRect(0, 0, w, h);
  // ... 绘图
}
```

### 不需要交互可视化的情况

纯硬件/纯流程类技术（如特定的样品前处理方法），无"参数→信号形态"因果链时，可省略，用 Mermaid 替代。

## 三、配套细化规范

> **本节是写作规范的"实施细则"**——给第一大节 / 第二大节的硬性要求提供具体的写作模板。
> 改规范时只动本节；不要在第一、二大节里复述本节内容。

### 3.1 公式撰写规范

| 要素 | 要求 | 示例 |
|:---|:---|:---|
| **公式本身** | LaTeX 行间公式 `$$...$$` | $$\Delta\tilde{\nu}_P = \frac{|\Delta\lambda_P|}{\lambda_P^2}$$ |
| **符号说明** | 公式下方列表说明每个符号 | $\lambda_P$：激发波长（nm） |
| **数值例子** | 代入典型参数算出数字 | $\lambda_P=785$ nm, $\Delta\lambda_P=0.5$ nm → $\Delta\tilde{\nu}_P \approx 8.1$ cm⁻¹ |
| **适用条件** | 标注公式成立的前提 | "瑞利区适用，μm 级细胞属 Mie 区，仅作机理理解" |

### 3.2 参考文献规范

- 每篇文献必须包含：作者、标题、期刊、年份、卷期页、DOI
- **DOI 必须核实**：用可用的网页浏览器工具（如 `web_fetch` 或 `web_search` 或 playwright MCP 的 `browser_xxx` 等）验证；无法核实的一律标注 `（待核实）`
- 分类排列：经典文献（按时间正序）→ 综述 → 应用论文
- 用户提供的应用论文排在应用类第一条
- **标注格式统一**：本节用 `（待核实）`；与第四章 §四 "诚实性红线" 第 2 条同名标注格式完全一致，便于跨章节检索与维护。

### 3.3 SVG 文件管理规范

1. **存放位置**：所有 SVG 一律放 `assets/`，文件名用**小写英文下划线**命名（例：`optical_trap_geometries.svg`），便于 git 跟踪与跨平台引用。
2. **在 md 中只引用、不内嵌**：
   ```markdown
   ![光路示意图（含受力箭头）](assets/optical_path.svg)
   ```
3. **跨平台一致渲染的写法约束**：
   - 避免 `xlink:href` 外链；
   - 字体用 `font-family="system-ui, -apple-system, 'Segoe UI', 'PingFang SC', sans-serif"` 兜底，**不用等宽字体强行对齐**；
   - 颜色用显式十六进制或 `currentColor`，不依赖外部 CSS class；
   - 文字标注用绝对像素坐标 + `text-anchor="middle/end"`，**不靠字符宽度对齐**。
4. **生成脚本也存档一份**（`scripts/generate_xx.{html,mjs,ts}`）：
   - **首推 .html**：理由见目录规范 callout（"scripts/ 首选 `.html` 的理由"）。
   - **.mjs / .ts**：仅当脚本逻辑重、不需要可视化结果、且团队有 Node 工具链时使用
   - **数据驱动 SVG**：`echarts` / `@antv/g2` / `D3.js`（CDN 引入即可）
   - **几何/示意 SVG**：手写即可（参考本 skill 的 `assets/optical_path.svg`）
   - **浏览器渲染转 SVG**：Playwright（headless 截图 → SVG）
   - **避免 Python matplotlib / Origin / gnuplot** —— 浏览器/手机/平板/GitHub preview 上 Python 产物无法直接复现
5. **文件命名示意**（建议风格）：
   - `optical_path.svg` 光路 / 几何示意
   - `trap_geometries.svg` 对比 point / donut / line 等几何形态
   - `spectrum_soc.svg` 数据谱图（与具体数据强相关）
   - `potential_well.svg` 势阱/势能面
   - `force_balance.svg` 受力分析

### 3.4 技术依赖处理规则（Avoid Pedagogical Drift）

#### 核心原则

**每个技术简报只覆盖其「主体技术」，不展开「基础依赖技术」的完整介绍。**

避免教学法漂移（pedagogical drift）：写 EC-SERS 时大段讲拉曼原理，写拉曼时又大段讲分子振动，链式蔓延会让单篇简报膨胀且失焦。

#### 定义

| 角色 | 含义 | 示例（主体=EC-SERS） |
|---|---|---|
| **主体技术 A** | 当前简报直接讨论的技术 | EC-SERS 联用 |
| **基础依赖 B** | A 的实现/理解所依赖的更底层技术 | 拉曼光谱、电化学基础 |
| **弱依赖 C** | 仅引用概念、不需读者深入理解 | LSPR、表面等离激元 |

#### 写作约束

**1. B 在 A 简报中的篇幅上限**

- **总篇幅 ≤ 10%** 或 **≤ 1 段**（取较严）
- 仅写"A 在什么环节用到了 B 的什么能力"，**不展开 B 的原理/历史/完整介绍**
- 给出关键术语 + 一句话定位 + 推荐查阅入口（教材章节/经典综述 DOI），让读者按需跳转

**2. 强依赖标注（简报末尾必填，对应 §一·15）**

完成主体内容后，在简报末尾添加「依赖地图」小节：

```
## 依赖地图（Dependency Map）

| 依赖强度 | 技术 | 在 A 中的角色 | 是否需要独立简报 |
|---|---|---|---|
| 🔴 强依赖 | 拉曼光谱基础 | 信号产生机理 | ⚠️ 待询问 |
| 🟡 弱依赖 | LSPR | 增强机理引用 | 否 |
| ⚪ 仅提及 | 电化学工作站 | 工具引用 | 否 |
```

**3. 询问而非默认展开**

简报交付时用以下句式询问用户（不自己决定）：

> 本简报在 **[环节]** 处强依赖 **[技术 B]**，但未展开其基础原理。
> 是否需要我单独撰写一份 B 的技术简报？
> （如需，请确认侧重方向：基础原理 / 工程实践 / 文献综述）

**4. 链式触发保护**

- 用户确认写 B 的简报后，**B 简报使用同一规则**，避免反向蔓延到 B 的更基础依赖（如拉曼的"分子振动"）。
- 每篇简报只在「依赖地图」中暴露**一层**依赖，深层依赖在 B 简报里再展开。

**5. 反例（避免的写法）**

❌ 在 EC-SERS 简报里花 30% 篇幅介绍拉曼散射的物理原理

❌ 把"基础铺垫"作为简报第一章（除非用户明确要求入门级）

❌ 不询问就主动展开 B 的独立章节（容易造成 A 简报失焦）

**6. 正例（鼓励的写法）**

### 3.5 Mermaid 踩坑清单

Mermaid 解析器对节点标签、边标签、箭头语法有严格的保留字 / 字符约束。下表为常见踩坑与规避方式：

| 现象 | 根因 | 规避 |
|:---|:---|:---|
| 节点标签 `V1["\|v=1⟩..."]` 解析失败 | `\|` 是边标签保留分隔符，引号内仍可能误识别 | Dirac 符号 `\|v⟩` 改用 `⟨v⟩`（U+27E8 / U+27E9）或省略 `\|` |
| `A -.->.` `==>>` `<<<` 等报错 | mermaid 箭头组合有固定白名单 | 严格用 `-->` `-.->` `==>` `<-->` `--x` 等固定写法 |
| `V1 -.->"\|label\|"\| VIRT` 报错 | 引号内已有 `\|`，又用 `\|` 当边标签分隔符（双重嵌套） | 引号 / `\|...\|` 包围两者选一 |
| `A -->\|text<br/>more\| B` 报错 | 边标签里 `<br/>` 未 quoted | 必须 `A -->\|"text<br/>more"\| B` |
| 节点 ID 与保留字冲突 | `subgraph` `end` `flowchart` 不能作为节点 ID | 用中文节点 ID（新版 mermaid 支持）或加前缀（`sub_` 等） |
| 子图名带括号 / 引号 | `subgraph "name"` 在某些版本失败 | 用 `subgraph name["..."]` 显式 ID + 标签 |
