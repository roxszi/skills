# 技术简报撰写 skill（tech-brief-writer）

### 何时用

用户要求为某项**技术/方法/仪器原理**撰写可长期存档的技术简报 / 备忘。**跨学科适用**——分析化学、物理化学、光谱学、化学计量学、生物、材料、半导体等任意技术领域均可。典型触发：

- "帮我写一个 XX 技术的技术简报，我存下来备忘"
- "看一下这篇文章用的技术，整理成档案"
- "把 XX 方法的原理 + 公式 + 应用整理一份 md"

### 何时不用

下列场景**不要**调用本 skill（避免误用）：

- 用户只是临时问"这是什么"，不要求存档 → 直接回答即可，不存档
- 用户要写的内容不是技术（如情感、生活、新闻、聊天） → 改用其它 skill 或不调
- 内容是文献综述 / 论文草稿而非技术备忘 → 改用论文写作 skill
- 内容是单点问答（如"XX 试剂用什么浓度"） → 直接回答，不存档

> ⚠️ **涉及基础依赖技术时的特别提示**：撰写前先判断是否属于"主体 + 依赖"复合技术（如 EC-SERS 主体 + 拉曼基础依赖）；若是，参见 §三·3.4 的技术依赖处理规则，避免教学法漂移（pedagogical drift）。

### 硬边界声明（独立 skill 的设计原则）

本 skill 明确**不绑死**以下内容，交给宿主环境决定：

- **具体学科领域**——任何技术领域都能用，§四 触发清单是跨学科通用骨架
- **具体技术栈**——DOI 抓取 / 浏览器自动化 / 编程语言等一律用"父级词汇"，具体工具按宿主实际选用
- **个人偏好**——表达风格、语言习惯、配套工具链等不沉淀进 skill
- **宿主业务**——教学场景、特定课题栈、OCR、私有文献库等不写进 skill

**这套边界是有意为之**——独立 skill 应能交付给任何宿主，而不是某个具体宿主的私有扩展。

### 目录规范

#### 默认父目录

简报统一放在工作区根目录下的 `tech-brief/` 子目录里（`tech-brief/<tech-name>/`）。

**用户覆盖规则**：用户在请求里**显式指定了其它位置**（如"写在 xxx 目录下"），按用户指定的来，不走默认。这一条优先于默认父目录——独立 skill 不应假设每个工作区都长一样。

#### 标准子目录结构

每个技术**独立一个文件夹**，结构如下：

```
<tech-name>/
├── 技术简报.md
├── 交互演示.html                        # 可选：交互演示
├── assets/                                # 可选：所有外部 SVG / 各类资源
│   ├── schematic.svg
│   ├── spectrum.svg
│   └── ...
├── scripts/                               # 可选：可执行 / 可复现的脚本（首推 .html）
│   ├── generate-xx.html                   # 浏览器内可直接打开：脚本即代码、结果即页面
│   ├── generate-xx.mjs                    # 可选：Node 端运行（需 bun/node 环境）
│   └── ...
└── README.md                              # 可选：目录索引（仅当文件数量较多 / 多简报共建 / 对外分享时）
```

#### 命名约定

| 文件类型 | 命名规范 | 说明 |
|:---|:---|:---|
| 子目录 | `<tech-name>/` | 放在父目录 `tech-brief/` 下；目录名已表达技术身份，无空格 |
| 简报 MD | `技术简报.md` | 目录已表达技术身份，文件名不再重复 |
| 交互演示 HTML | `交互演示.html` | 目录已表达技术身份，文件名不再重复 |
| 生成脚本 | `scripts/generate-xx.{html,mjs,ts}` | kebab-case（web 习惯） |
| SVG 资源 | `assets/<用途>.svg` | 纯 kebab-case（web 习惯）；`<用途>` 复合描述，目录已表达技术身份 |

> 📁 **scripts/ 首选 `.html` 的理由**：
> - 浏览器 = 运行时 + IDE + 查看器，**三者合一**；
> - 所有设备（PC / 平板 / 手机）都有浏览器，零环境依赖；
> - 文件即程序：`<script>` 里的代码就是脚本本体，保存即生效（刷新即看结果）；
> - HTML 在这里**兼有"脚本"和"产物"双重身份**——既是 `generate-xx.html`，也可以直接拷贝关键 SVG 到 `assets/` 落档。

## 一、MD 简报章节模板

按需裁剪，但**加粗项 / ★ 项为硬性要求**。

> 📌 **章节模板 vs 配套细化规范**：本章是"每节里要写什么"的硬性要求骨架；具体写作格式（公式、参考文献、SVG、依赖处理）单列为第三大节的细化规范，避免与本章互相重复。改规范只动对应章节。
>
> 📌 **依赖地图必填**：本模板第 13 章（依赖地图）是有跨章节生效的依赖类技术的硬性产物，详见 §三·3.4。
> 
> 📌 **formatter 元数据必填**：markdown 顶部的 yaml formatter 元数据（文档性质 / 最后修订日期 / 作者 / 发起人 / 其它有必要写在文档头的内容）。
> 
> 📌 **目录必填**：位置在 markdown 标题之下，§1 之上。

1. **一句话定义**：blockquote 引用块开篇（中文定义 + 英文全称/缩写 + 典型应用场景）
2. **核心技术原理**
   - 问题背景（可用"传统方案 vs 局限"对比表引入痛点）
   - ★ **一个统一核心洞察（single conceptual anchor）**：把整项技术的精髓凝练成**一张表或一句话**（例：SERDS 的"坐标对偶性"——谁不动谁被差分消掉；LTRS 的"梯度力 vs 散射力平衡"）。这是整篇的灵魂锚点，读者记住这一个就抓住了本质。
3. **原理示意图**：**§3.1 Mermaid（流程 / 因果图）** + **§3.2 外置 SVG（几何 / 谱图）**；二者互补、**禁止用 ASCII art 替代几何 / 谱图**
   - **§3.1 Mermaid**：算法步骤 / 数据流 / 因果链。Mermaid 语法踩坑清单见 §三·3.5
   - **§3.2 外置 SVG**：频谱位置、χ⁽³⁾ 谱形、信号路径、光路几何等几何 / 谱图类内容
     - 一律外置到 `assets/` 下，md 用标准图片语法引用：
       ```markdown
       ![光阱结构示意图](assets/trap-geometries.svg)
       ```
     - **不内嵌** `<svg>` 块到 md 里（部分 markdown 渲染器对 inline SVG 支持不一致）
     - 完整规范（命名 / 字体 / 锚点 / 颜色 / 生成脚本）见 §三·3.3
   - **★ 禁止用 ASCII art（```代码块围栏的字符图）替代几何 / 谱图**：等宽字体宽度不一导致错位、emoji 与中文混排错位、不可缩放、无颜色编码
4. **基本公式方程**
   - 每个关键公式后尽量给 **带入数值的算例（worked example）**（如 "λ=785 nm, Δλ=0.5 nm → Δν̃≈8.1 cm⁻¹"）
   - ★ **适用区间/有效性警示**：注明公式成立的前提与边界（如瑞利区 vs Mie 区、Kasha 规则例外）。**不给未经数学/物理验证的式子，边界必须写清**
   - 公式四要素（公式本体 / 符号说明 / 数值例子 / 适用条件）的统一写法见 §三·3.1。
5. 技术变体 / 发展（可选）：该技术存在 ≥1 个主流变体时必填；否则省略。同样配图 + 公式 + 对比表。
6. **数据处理算法**（涉及机器学习 / 统计建模 / 化学计量学 / 信号处理等数据驱动方法时**必做**）：算法 / 原理 / 优势 / 适用场景 四列表。
7. **典型示例应用**
   - 论文信息：标题、期刊年份、**链接 + DOI**
   - 仪器配置表、样品集、处理策略、关键发现
   - ★ **明确标注"用到本技术/变体的具体位置"**（便于核对"本技术是否真在论文中被使用"）
8. **注意事项与工程要点**：⚠️ **踩坑框** + 表格（"要点 / 原因"两列）
9. **优缺点**（优点 / 缺点约束，分列）
10. **参考文献**（末尾必填）：每篇文献必须包含作者 / 标题 / 期刊 / 年份 / 卷期页 / DOI；DOI 必须核实，未核实的一律标注 `（待核实）`；分类排列：经典（按时间正序）→ 综述 → 应用论文；用户提供的应用论文排应用类第一条。**完整规范见 §三·3.2。**
11. 记忆口诀（可选但推荐）：当存在多个核心公式 / 概念需要记住时，一句话助记。
12. **溯源脚注（provenance footer）**：斜体注明"基于谁在何时的什么要求或什么样的讨论进行整理/修订"；**若过程中纠正过错误认知，明确记录下来**（诚实透明，也便于日后追溯）
13. **依赖地图（Dependency Map，末尾必填）**：以一张四列表（依赖强度 / 技术 / 在 A 中的角色 / 是否需要独立简报）声明本简报与外部基础技术的关系。格式与判定准则见 §三·3.4。

> 💡 **填空骨架（template）**：本 skill 同目录下的 [`template.md`](template.md) 与上述 13 章节列表 **1-to-1 对齐**，以 `{占位符}` 形式给出可直接套用的空白简报结构。
> 
> **推荐工作流**：
> 1. 复制 `template.md` → 重命名为 `技术简报.md`
> 2. 按 §一 顺序，逐章替换 `{占位符}` 内容
> 3. 按需裁剪可选章节（目录 / 技术变体 §5 / 记忆口诀 §11 等）
> 4. 完成后走 §五 自检清单

## 二、演示（交互式 HTML）规格

### 触发条件

当技术的核心原理涉及**"参数变化 → 光谱/信号形态变化"**等因果关系时（例如：波长差影响差分峰幅度、光阱结构影响探测区域），且能用几个滑块让核心物理量变化趋势可视化时，**必须**生成一份独立 HTML 交互演示。

### 技术要求

- **单文件自包含**：HTML + JS + `<canvas>`，**离线双击可开**
- **技术栈**：JS/TS 优先，ESM 风格；若涉及张量 / ML 再考虑 tfjs（浏览器端）
- **中文页面**：标题、轴标签、图例全部中文
- **高 DPI 适配**：读 `window.devicePixelRatio`，canvas 物理像素 = CSS 尺寸 × dpr
  - ⚠️ **ImageData 索引必须用整数（imgData.width / imgData.height），不能用浮点 plotW / plotH**。否则会产生 小数索引，被自动 `Math.floor` 后落到错误像素的 A 通道，导致热图/位图大面积失色。
  - `canvas.width/height/style.width/height` 任何时候都不在重绘函数里赋值。违反 = 物理像素累积爆炸。
- **参数滑块实时重算重绘**：
  - 简单交互（默认）：`input` 事件 → 重新 `computeData()` → `drawAll()`，vanilla JS 即可，**零依赖**
  - 较复杂交互响应（多 view、组件化、跨滑块共享状态）：**VUE3 渐进式引入** —— 以 CDN 单文件形式挂载，**不要**走 `npm + Vite/webpack`：
    ```html
    <script src="https://unpkg.com/vue@3/dist/vue.global.js"></script>
    ```
    ⚠️ CDN 引入意味着**首次加载需联网**（`vue.global.js` 浏览器缓存后即可离线）；因此务必**在 HTML 中显式声明 "需要联网"**，让用户知道。
- **围绕"核心洞察"设计对比视图**：配色与内容含义匹配（正确用暖绿/错用暖红等），一眼看懂
- **中文注释**：物理参数、函数、算法流程均中文注释
- **物理要忠实**：演示里的公式实现要和 MD 里的公式一致，不能为了好看而失真
- 自绘坐标轴/网格/图例/零线；峰型用洛伦兹（拉曼峰）/高斯（荧光/带）等符合物理的函数
- 若演示中也用到同样的几何/谱图示意（**非交互态**），直接 **`assets/` 单一来源复用** SVG（详见 §三·3.3 第 4 条），避免在演示里用 canvas 重画一遍静态图。
- **交付前自测**（硬性要求）：演示 HTML 完成后，**必须**在本地浏览器打开、逐个交互 ≥ 1 次、运行 ≥ 30 秒，目视无 bug 后才能提交。常见 bug 清单：
  - 选参数时画面**累积变大/变小**（高 DPI 像素累积陷阱）
  - 滑块拖动时画面**不响应**（事件未绑定 / state 未更新）
  - 循环动画（CV 扫描）**越来越快**（`requestAnimationFrame` 累积未取消）
  - 切换下拉时**canvas 残留**（清屏时未用 dpr 重置坐标系）

### 高 DPI Canvas 模板

```js
// 初始化（页面加载时一次性执行）
function initCanvas(canvas) {
  const dpr = window.devicePixelRatio || 1;
  const cssW = canvas.clientWidth || canvas.width;  // CSS 像素
  const cssH = canvas.clientHeight || canvas.height;
  canvas.width  = cssW * dpr;   // 物理像素：只设这一次
  canvas.height = cssH * dpr;
  canvas.style.width  = cssW + "px";
  canvas.style.height = cssH + "px";
  canvas._ctx = canvas.getContext("2d");
  canvas._dpr = dpr;
  canvas._cssW = cssW; canvas._cssH = cssH;
}
// 重绘（每次更新调用）
function redraw(canvas) {
  const { _ctx: ctx, _dpr: dpr, _cssW: w, _cssH: h } = canvas;
  ctx.setTransform(dpr, 0, 0, dpr, 0, 0);   // 只重置坐标系
  ctx.clearRect(0, 0, w, h);
  // ... 绘图
}
```

### Canvas 散点可视化反模式（必读）

演示中大量场景需要画散点（标定曲线、重复测量、PCA 投影等）。**绝对避免的反模式**：

#### 反模式：用 `Math.random()` 给散点的位置生成随机偏移

```js
// ❌ 错误：每次重绘都重新随机化
for (let i = 0; i < N; i++) {
  const x = baseX + (Math.random() - 0.5) * 2;  // 每次都不同
  const y = baseY + (Math.random() - 0.5) * 2;
  draw(x, y);
}
```

**现象**：用户拖动噪声滑块时画面"看起来完全不同的散点"——实际是变化了，但散点位置整体漂移，看不到宽度变化。

#### 正确做法：Mulberry32 seeded PRNG + 固定 SCATTER_OFFSETS

```js
// Mulberry32 PRNG
function mulberry32(seed) {
  return function () {
    seed = (seed + 0x6D2B79F5) | 0;
    let t = seed;
    t = Math.imul(t ^ (t >>> 15), t | 1);
    t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  };
}

// 模块级预生成固定偏移（页面加载时一次性）
const SCATTER_OFFSETS = (() => {
  const rng = mulberry32(20260711);  // 任意固定 seed
  const arr = [];
  for (let i = 0; i < N; i++) arr.push((rng() - 0.5) * range);
  return arr;
})();

// 重绘时：c 偏移用 SCATTER_OFFSETS（位置稳定），噪声用确定性方向 × 当前 noiseSigma
for (let i = 0; i < N; i++) {
  const noiseDir = (i % 2 === 0 ? 1 : -1) * ((i * 0.37) % 1 - 0.5) * 2;
  const noiseSample = noiseDir * state.noiseSigma;  // 来自滑块
  const c = baseConc + SCATTER_OFFSETS[i];           // 位置稳定
  const A = slope * c + noiseSample;                  // Y 散布随噪声变化
  draw(c, A);
}
```

**验证方法**：调噪声滑块时，散点的 **X 轴位置不变**，只 **Y 轴散布宽度变化**。两者都变 = SCATTER_OFFSETS 没生效或被覆盖。

#### 配套可视化（强烈推荐）

为让"噪声影响"一眼可见，加 **±σ 误差带**（半透明矩形 + 虚线边界）作为辅助层。

### 不需要交互可视化的情况

#### 可省略 HTML 演示（满足 ≥ 1 即省略）

- 技术核心无"参数 → 信号形态"因果链
- 技术是 SOP / 制备方法 / 维护流程
- 技术核心是单点公式或常数表
- 用 Mermaid / SVG 已经能完整传达原理

#### 必须 HTML 演示（满足 ≥ 1 即必须）

- 至少 1 个滑块能改变谱形 / 几何 / 信号
- 演示能体现 §2.2 灵魂锚点的对比视图（"正确 vs 错误"、"A vs B"等）
- 演示能让读者"看到某个 trade-off 怎么动"
- 核心机理涉及多参数协同（如 §4 例子中"捕获 vs 激发功率 trade-off"）

> **判据不互斥时**：若一项满足"必须"，按必须做；否则按"可省略"判断。**默认优先用 Mermaid + SVG**——只在 Mermaid 表达不清时升级到 HTML 演示。

## 三、配套细化规范

> **本节是写作规范的"实施细则"**——给第一大节 / 第二大节的硬性要求提供具体的写作模板。
> 改规范时只动本节；不要在第一、二大节里复述本节内容。

### 3.1 公式撰写规范

| 要素 | 要求 | 示例 |
|:---|:---|:---|
| **公式本身** | LaTeX 行间公式 `$$...$$` | $$\Delta\tilde{\nu}_P = \frac{|\Delta\lambda_P|}{\lambda_P^2}$$ |
| **符号说明** | 公式下方列表说明每个符号 | $\lambda_P$：激发波长（nm） |
| **数值例子** | 代入典型参数算出数字 | $\lambda_P=785$ nm, $\Delta\lambda_P=0.5$ nm → $\Delta\tilde{\nu}_P \approx 8.1$ cm⁻¹ |
| **适用条件** | 标注公式成立的前提 | "瑞利区适用，μm 级细胞属 Mie 区，仅作机理理解" |

### 3.2 参考文献规范

#### 3.2.1 本技术文献库反查约定（写之前先查）

引用任何文献之前，**先查宿主已有的本技术文献库**。具体库类型 / 工具视宿主环境而定，常见形式包括但不限于：

- 基于 SQLite 的本地库（命令行工具）
- 基于 Markdown 文件的目录式库
- Zotero / EndNote / Mendeley 等文献管理软件
- Notion / Obsidian 等笔记型库
- 任意自定义的本地检索系统

无论库的具体形态如何，反查约定一致：

| 步骤 | 必须做的 |
|:---|:---|
| 1. 检索 | 用库自带的检索方式查找（**中文检索与英文检索可能使用不同机制**，如 FTS 与 LIKE、unicode 分词差异等，按宿主工具实际能力选用） |
| 2. 取全文 | 命中后**必须从库内取全文再讲解**，不重新 fetch，不靠摘要脑补 |
| 3. 同义词扩展 | 同义词 / 别名 / 缩写需手动 OR（如 `Raman OR 拉曼`）——多数 FTS 引擎不会自动扩展 |
| 4. 未命中 | 才走外部抓取（§3.2.2 决策树） |

> 📌 **独立 skill 不绑死库的具体形态**：上表是跨库通用的"先查 → 取全文 → 讲解"骨架。具体命令 / API 按宿主实际工具选用。

#### 3.2.2 文献条目规范

- 每篇文献必须包含：作者、标题、期刊、年份、卷期页、DOI
- **DOI 必须核实**：用宿主可用的**网页抓取类工具**（具体可用工具视宿主环境而定，包括但不限于 `web_fetch` / `web_search` 类工具、浏览器自动化类工具、命令行 `curl` 等）验证；无法核实的一律标注 `（待核实）`
- 分类排列：经典文献（按时间正序）→ 综述 → 应用论文
- 用户提供的应用论文排在应用类第一条
- **标注格式统一**：本节用 `（待核实）`；与第四章 §四 "诚实性红线" 第 2 条同名标注格式完全一致，便于跨章节检索与维护。

#### DOI / 全文获取决策树（父级词汇）

独立 skill **不绑死具体技术栈**——具体用什么抓取工具看宿主环境。下面的决策树用"父级词汇"组织，任意宿主都能套用：

```
DOI / URL 给定
  │
  ├─ 步骤 1：用宿主可用的网页抓取类工具一次尝试（如 fetch 类 / web_fetch 类 / search 类工具等，具体可用工具按宿主环境而定）
  │   ├─ 成功且拿到完整正文 → 走 §三·3.2 后续规范引用即可
  │   └─ 失败（403 / 反爬 / 截断）→ 进入步骤 2
  │
  ├─ 步骤 2：换另一种抓取类工具（如 fetch 类失败换浏览器自动化类工具）
  │   ├─ 成功 → 走 §三·3.2 后续规范引用即可
  │   └─ 失败 → 进入步骤 3
  │
  ├─ 步骤 3：尝试备用入口
  │   ├─ 作者主页 / 预印本平台 / 开放获取镜像 / 机构知识库
  │   ├─ 成功 → 走 §三·3.2 后续规范引用即可
  │   └─ 失败 → 进入步骤 4
  │
  └─ 步骤 4：标注 `（待核实）` 并在溯源脚注里说明尝试过的入口
```

**关键原则**：
- **不要在第一步失败就放弃**——按决策树至少尝试到步骤 3
- **不要为了"看起来完整"伪造成功**——失败就标 `（待核实）`，诚实优于体面
- **抓到的内容是 JSON 包装（多模态抓取类工具常见格式）** → 走 `clean.ts --from-mcp` 类清洗工具后人工 review（具体清洗工具名按宿主实际可用工具选用）

### 3.3 SVG 文件管理规范

1. **存放位置**：所有 SVG 一律放 `assets/`，文件名用**纯 kebab-case**（web 习惯）命名（例：`trap-geometries.svg`），便于 git 跟踪与跨平台引用。
2. **在 md 中只引用、不内嵌**：
   ```markdown
   ![光路示意图（含受力箭头）](assets/optical-path.svg)
   ```
3. **跨平台一致渲染的写法约束**：
   - 避免 `xlink:href` 外链；
   - 字体用 `font-family="system-ui, -apple-system, 'Segoe UI', 'PingFang SC', sans-serif"` 兜底，**不用等宽字体强行对齐**；
   - 颜色用显式十六进制或 `currentColor`，不依赖外部 CSS class；
   - 文字标注用绝对像素坐标 + `text-anchor="middle/end"`，**不靠字符宽度对齐**。
4. **脚本是否需要存档 & 生成 SVG 的几种方式**：
   
   **a) 脚本类型与存档**（命名沿用目录规范的 `scripts/generate-xx.{html,mjs,ts}`）：
   - **首推 .html**：理由见目录规范 callout（"scripts/ 首选 `.html` 的理由"）。
   - **.mjs / .ts**：仅当脚本逻辑重、不需要可视化结果、且团队有 Node 工具链时使用
   
   **b) 生成 SVG 的几种方式**：
   - **几何 / 示意 SVG**：手写即可（参考本 skill 的 `assets/optical-path.svg`）。
   - **数据驱动 SVG**：`echarts` / `@antv/g2`（CDN 引入即可，需设置好默认输出 SVG 而非 canvas）；**D3.js**（天然输出 SVG DOM，零配置）。
   - **浏览器渲染转 SVG**：Playwright（headless 截图 → PNG/SVG；PNG 需经 svgtrace 等工具栅矢量化）。
   - **避免 Python matplotlib / Origin / gnuplot** —— 浏览器/手机/平板/GitHub preview 上 Python 产物无法直接复现。
5. **文件命名示意**（建议风格）：
   - `optical-path.svg` 光路 / 几何示意
   - `trap-geometries.svg` 对比 point / donut / line 等几何形态
   - `spectrum-soc.svg` 数据谱图（与具体数据强相关）
   - `dual-wavelength-spectrum.svg` 双波长光谱示意
   - `potential-well.svg` 势阱 / 势能面
   - `force-balance.svg` 受力分析

### 3.4 技术依赖处理规则（Avoid Pedagogical Drift）

#### 核心原则

**每个技术简报只覆盖其「主体技术」，不展开「基础依赖技术」的完整介绍。**

避免教学法漂移（pedagogical drift）：写 EC-SERS 时大段讲拉曼原理，写拉曼时又大段讲分子振动，链式蔓延会让单篇简报膨胀且失焦。

#### 定义

| 角色 | 含义 | 示例（主体=EC-SERS） |
|---|---|---|
| **主体技术 A** | 当前简报直接讨论的技术 | EC-SERS 联用 |
| **基础依赖 B** | A 的实现/理解所依赖的更底层技术 | 拉曼光谱、电化学基础 |
| **弱依赖 C** | 仅引用概念、不需读者深入理解 | LSPR、表面等离激元 |

#### 依赖强度判据（看到就判定）

判别某项依赖属于哪一档，**满足任一即按该档处理**：

| 强度 | 判据（满足任一即归此类） |
|:---|:---|
| **🔴 强依赖** | ① A 的 §2 原理叙述（不含灵魂锚点）必须靠 B 才能讲清楚<br>② A 的公式里有 B 的物理量作为自变量（且不是直接引用）<br>③ A 的实验结果里反复出现 B 的标志性现象 / 术语<br>④ A 的 §2.2 灵魂锚点的关键推论 / 数据来源必须靠 B 才能成立 |
| **🟡 弱依赖** | ① 仅在 A 的某一变体或某一章节涉及 B<br>② A 引用 B 作为补充解释，但不展开 |
| **⚪ 仅提及** | ① A 仅引用 B 作为工具 / 实验条件 / 标准件<br>② B 的存在对理解 A 没有影响 |

> **视角差异**：判据 ① 与 ④ 在多数场景下重合（前者描述 §2 原理叙述，后者锁定 §2.2 灵魂锚点）；视为任一即判定 🔴 强依赖即可。

#### 写作约束

**1. B 在 A 简报中的篇幅上限**

- **总篇幅 ≤ 10%** 或 **≤ 1 段**（取较严）
- 仅写"A 在什么环节用到了 B 的什么能力"，**不展开 B 的原理/历史/完整介绍**
- 给出关键术语 + 一句话定位 + 推荐查阅入口（教材章节/经典综述 DOI），让读者按需跳转

**2. 强依赖标注（简报末尾必填，对应 §13 依赖地图）**

完成主体内容后，在简报末尾添加「依赖地图」小节：

```
## 依赖地图（Dependency Map）

| 依赖强度 | 技术 | 在 A 中的角色 | 是否需要独立简报 |
|---|---|---|---|
| 🔴 强依赖 | 拉曼光谱基础 | 信号产生机理 | ⚠️ 待询问 |
| 🟡 弱依赖 | LSPR | 增强机理引用 | 否 |
| ⚪ 仅提及 | 电化学工作站 | 工具引用 | 否 |
```

**3. 询问而非默认展开**

简报交付时用以下句式询问用户（不自己决定）：

> 本简报在 **[环节]** 处强依赖 **[技术 B]**，但未展开其基础原理。
> 是否需要我单独撰写一份 B 的技术简报？
> （如需，请确认侧重方向：基础原理 / 工程实践 / 文献综述）

**4. 链式触发保护**

- 用户确认写 B 的简报后，**B 简报使用同一规则**，避免反向蔓延到 B 的更基础依赖（如拉曼的"分子振动"）。
- 每篇简报只在「依赖地图」中暴露**一层**依赖，深层依赖在 B 简报里再展开。

**5. 反例（避免的写法）**

❌ 在 EC-SERS 简报里花 30% 篇幅介绍拉曼散射的物理原理

❌ 把"基础铺垫"作为简报第一章（除非用户明确要求入门级）

❌ 不询问就主动展开 B 的独立章节（容易造成 A 简报失焦）

**6. 正例（鼓励的写法）**

✅ EC-SERS 简报里写："信号增强依赖于拉曼散射（基础原理见 [依赖地图]，本文不展开）"

✅ 在简报末尾列出强依赖并询问

✅ 用户问"拉曼是什么"时再触发 B 简报的撰写

### 3.5 Mermaid 踩坑清单

Mermaid 解析器对节点标签、边标签、箭头语法有严格的保留字 / 字符约束。下表为常见踩坑与规避方式：

| 现象 | 根因 | 规避 |
|:---|:---|:---|
| 节点标签 `V1["\|v=1⟩..."]` 解析失败 | `\|` 是边标签保留分隔符，引号内仍可能误识别 | Dirac 符号 `\|v⟩` 改用 `⟨v⟩`（U+27E8 / U+27E9）或省略 `\|` |
| `A -.->.` `==>>` `<<<` 等报错 | mermaid 箭头组合有固定白名单 | 严格用 `-->` `-.->` `==>` `<-->` `--x` 等固定写法 |
| `V1 -.->"\|label\|"\| VIRT` 报错 | 引号内已有 `\|`，又用 `\|` 当边标签分隔符（双重嵌套） | 引号 / `\|...\|` 包围两者选一 |
| `A -->\|text<br/>more\| B` 报错 | 边标签里 `<br/>` 未 quoted | 必须 `A -->\|"text<br/>more"\| B` |
| 节点 ID 与保留字冲突 | `subgraph` `end` `flowchart` 不能作为节点 ID | 用中文节点 ID（新版 mermaid 支持）或加前缀（`sub_` 等） |
| 子图名带括号 / 引号 | `subgraph "name"` 在某些版本失败 | 用 `subgraph name["..."]` 显式 ID + 标签 |

#### Mermaid 渲染验证 SOP

写完 Mermaid 后**必须验证**，不能交付未经验证的图：

1. **本地预览**：用宿主可用的 Markdown 预览工具（VSCode + Markdown Preview Enhanced / Typora / Obsidian 等，具体按宿主实际选用）打开 MD 文件，确认 Mermaid 块渲染无报错。
2. **逐项对照踩坑清单**：把上表的 6 条逐项扫一遍，特别注意 `\|` / 箭头组合 / 节点 ID / `<br/>` 这四类高频坑。
3. **失败定位**：若渲染失败，根据报错定位到具体行号 → 在原 MD 的 Mermaid 块中标注问题字符 → 修改 → 重新预览。
4. **跨平台一致性**：Mermaid 版本不一致会导致同一段代码在不同环境渲染不同。建议在简报末尾溯源脚注里注明使用的 Mermaid 版本范围（如"v10+ 推荐"）。

## 四、诚实性红线（不可破）

### 4.1 核心四条

- **公式、理论，均必须得有出处、依据、或参考文献**，否则不写
- **不确定就标注**：DOI / 路径 / 包名 / API 等"硬事实"未核实时标 `（待核实）`，不靠记忆脑补
- **数学验证底线**：任何拟合公式 / 数学模型都**必须有数学或物理来源、或经过验证**。找不到来源的拟合不写；"应该差不多"不构成理由。这条不可破例（独立 skill 不绑死具体学科——任何技术领域的简报都适用）
- **路径/文件名先用工具核实**再下笔（Read/Bash 等），不靠记忆脑补

### 4.2 通用触发式自查清单（看到就触发）

下表是**通用学术诚信规则**，不绑死任何具体学科。撰写涉及下列内容的章节时，必须自查：

| 触发信号 | 必须做的事 |
|:---|:---|
| 看到 **p < 0.05** 或"显著差异" | 追问 n 是多少？效应量呢？样本是否独立？ |
| 看到 **R² > 0.99** 或"完美拟合" | 反问残差图过没过？过拟合风险评估了吗？ |
| 涉及**方法学验证 / 性能确认**类任务 | 按通用框架走：专属性 / 线性 / LOD/LOQ / 准确度 / 精密度 / 鲁棒性（顺序一个不能少） |
| **LOD/LOQ 数值** | 用 S/N = 3 / 10 等明确标准方法计算，**不能目测外推** |
| **数据与文献不符** | 深挖机理（这是研究机会，不是误差），不要硬调和 |
| **增强因子 / 灵敏度 / 检出限类比值** | 公式完整写出，给出全部来源数据，不省略分母 |
| **DFT / 仿真类数值** | 注明 functional / basis set / 收敛阈值 |
| **体外释放度 / 动力学曲线** | 注明介质 / 温度 / 转速 / 取样时间点等关键实验条件 |

> 📌 **独立 skill 不绑死具体学科**：上表是跨学科通用的"看到 X → 必须做 Y"格式。具体应用时（如物理化学、生物、材料、半导体等），按对应领域的对应规则展开。skill 不替使用者写完所有领域的规则——只给"什么时候必须自查"这个骨架。

### 4.3 找不到来源/全文/数据争议时的处置

| 情况 | 处置 |
|:---|:---|
| DOI 验证失败（抓取工具全部失败） | 标注 `（待核实）`；在溯源脚注里说明尝试过的入口 |
| 找不到全文（仅有摘要） | 标注 `（仅摘要）`；引用时**仅引用摘要明确陈述的结论**，不脑补推论 |
| 数据来源有争议 | 列出多方来源 + 各自表述 + 标注差异，不擅自调和 |
| 算例的数值算出来跟文献不符 | **不要硬调和**——按 §4.2 第 5 行"数据与文献不符 → 深挖机理"处理，记录在溯源脚注 |
