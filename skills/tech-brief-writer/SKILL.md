# 技术简报撰写 skill（tech-brief-writer）

### 触发条件

**何时用**：用户要求为某项技术/方法/仪器原理撰写**可长期存档**的技术简报 / 备忘。**跨学科适用**——分析化学、物理化学、光谱学、化学计量学、生物、材料、半导体等任意技术领域均可。典型触发：

- "帮我写一个 XX 技术的技术简报，我存下来备忘"
- "看一下这篇文章用的技术，整理成档案"
- "把 XX 方法的原理 + 公式 + 应用整理一份 md"

**何时不用**：

- 临时问"这是什么"，不要求存档 → 直接回答
- 内容不是技术（情感 / 生活 / 新闻 / 聊天）→ 改用其它 skill
- 文献综述 / 论文草稿而非技术备忘 → 改用论文写作 skill
- 单点问答（如"XX 试剂用什么浓度"）→ 直接回答

> ⚠️ **涉及基础依赖技术时**：撰写前先判断是否属于"主体 + 依赖"复合技术（如 EC-SERS 主体 + 拉曼基础依赖）；若是，参见 §三·3.4 的技术依赖处理规则，避免教学法漂移（pedagogical drift）。

> 🧱 **硬边界（独立 skill 设计原则）**：本 skill 不绑死 ① 具体学科领域（跨学科通用）、② 具体技术栈（DOI 抓取 / 浏览器自动化 / 编程语言等用"父级词汇"）、③ 个人偏好（表达风格 / 语言习惯 / 配套工具链）、④ 宿主业务（教学场景 / 课题栈 / OCR / 私有文献库）。这套边界**有意为之**——独立 skill 应能交付给任何宿主。

### 目录规范

#### 默认父目录

简报统一放在工作区根目录下的 `tech-brief/` 子目录里（`tech-brief/<Tech-Name>/`）。

**用户覆盖规则**：用户在请求里**显式指定了其它位置**（如"写在 xxx 目录下"），按用户指定的来，不走默认。这一条优先于默认父目录——独立 skill 不应假设每个工作区都长一样。

#### 标准子目录结构

每个技术**独立一个文件夹**，结构如下：

```
<Tech-Name>/
├── 技术简报.md
├── 交互演示.html                        # 可选：交互演示
├── assets/                                # 可选：所有外部 SVG / 各类资源
│   ├── schematic.svg
│   ├── spectrum.svg
│   └── ...
├── scripts/                               # 可选：可执行 / 可复现的脚本（首推 .html）
│   ├── generate-xx.html                   # 浏览器内可直接打开：脚本即代码、结果即页面
│   ├── generate-xx.mjs                    # 可选：Node 端运行（需 bun/node 环境）
│   └── ...
└── README.md                              # 可选：目录索引（仅当文件数量较多 / 多简报共建 / 对外分享时）
```

#### 命名约定

| 文件类型 | 命名规范 | 说明 |
|:---|:---|:---|
| 子目录 | `<Tech-Name>/` | 放在父目录 `tech-brief/` 下；目录名已表达技术身份，无空格 |
| 简报 MD | `技术简报.md` | 目录已表达技术身份，文件名不再重复 |
| 交互演示 HTML | `交互演示.html` | 目录已表达技术身份，文件名不再重复 |
| 生成脚本 | `scripts/generate-xx.{html,mjs,ts}` | kebab-case（web 习惯） |
| SVG 资源 | `assets/<用途>.svg` | 纯 kebab-case（web 习惯）；`<用途>` 复合描述，目录已表达技术身份 |

> 📁 **scripts/ 首选 `.html` 的理由**：
> - 浏览器 = 运行时 + IDE + 查看器，**三者合一**；
> - 所有设备（PC / 平板 / 手机）都有浏览器，零环境依赖；
> - 文件即程序：`<script>` 里的代码就是脚本本体，保存即生效（刷新即看结果）；
> - HTML 在这里**兼有"脚本"和"产物"双重身份**——既是 `generate-xx.html`，也可以直接拷贝关键 SVG 到 `assets/` 落档。

### 受众（`技术简报.md` yaml `受众` 字段）

- **用户指定了具体受众/读者画像** → 用用户给的（优先于默认）
- **用户未指定** → 默认受众：「**技术简报所在领域的平均知识水平的刚入学研究生**」——熟悉该领域本科基础课 + 入门研究方法（基本谱学 / 电化学 / 数据处理基础等）；不熟悉超出研究生入门课程的各专业主题（具体阈值由简报主题所属领域推断）。
- 用户主动说"我不懂 X" / "我不熟悉 Y" 时，必须在受众字段里明确补写 "不熟悉 X / Y"（详见 §3.1「问题背景」的「前置概念」触发规则）

## 一、MD 简报章节模板

按需裁剪，但**加粗项 / ★ 项为硬性要求**。

> 📌 **章节模板 vs 配套细化规范**：本章是"每节里要写什么"的硬性要求骨架；具体写作格式（公式、参考文献、SVG、依赖处理）单列为 §三 细化规范。
>
> 📌 **正文章节外的2项必填项**：
> 1. markdown 顶部 **yaml formatter**（文档性质 / 最后修订日期 / 作者 / 发起人 / 受众）；
> 2. 标题之下、§1 之上的**目录**；

1. **一句话定义**：blockquote 引用块开篇（中文定义 + 英文全称/缩写 + 典型应用场景）
2. **术语/名词表** ★（独立一级标题章节）：
   - **§2.1 专有名词 + 公式物理量符号**：统一解释全文涉及到的所有术语。**两类必须全覆盖**：
     - **专有名词**（名称 / 缩写）——含义 + 首次正式定义章节位置
     - **公式物理量符号**（Γ / $E_R$ 等符号类物理量）——含义 + 数值/常用取值 + 首次正式定义章节位置
     - **诊断信号**：用户读完 §3.1 反馈“什么是穆斯堡尔核”=专有名词未铺；反馈“Γ是啥”=物理量符号未铺
   - **§2.2 同义术语与别名**：选一个名称全文统一使用；其它别名仅在术语表里列出**一次**，正文**严禁别名混用**（反面：正文时而用“乙酸”时而用“冰醋酸”，等于给读者设置阅读门槛）
   - **§2.3 意思相近术语 + 分类标准歧义术语**：**意思相近术语**（同源但有明确区别，无法彼此替代）必须**明确区分**用法差异（典型如γ射线 vs X射线——同源都是高能光子，但按起源分类时不可替代；按能量分类时 10-100 keV 区间两者兼有）；**分类标准歧义术语**必须说明分类依据（如"光子"按起源 / 能量 / 波长分类时归属不同）；**其他需显式区分说明的术语**（如同一技术在不同学科语境下的不同侧重）
3. **核心技术原理**
   - **§3.1 技术背景**。**必须**遵循三段叙事顺序：
       0. **前置概念**：当简报内容判定涉及 yaml `受众` 可能不熟悉的前置概念（核物理 / 量子场论 / 多普勒能谱 / 群论 / 张量代数 等）时，**必须**在 §3.1「问题背景」开头加「📖 前置概念」部分——用读者已熟悉的基础概念做类比锚点（如电子跃迁 → γ 跃迁），判定条件（满足任一即触发）：① 用户口头说"我不懂 / 我不熟悉 X"；② yaml `受众` 字段明确写了"不熟悉 XXX"；③ 内容/术语或涉及的概念明显超出 yaml `受众` 正常的熟悉范围。
       1. **Happy path**：先讲该技术的"成功路径"长什么样（含 1 句话点明好处 / 意义）——让读者建立"为什么这件事值得做"的认知基线
       2. **Fail / 反直觉**：再讲为什么在某种条件下会失败 / 反直觉——读者有了 happy path 基线才能理解"失败的物理图景"
       3. **Fix / 解法**：再讲该技术如何解决 Fail——首尾呼应 happy path
       - **三段缺一不可，顺序不可颠倒**——这是技术简报区别于"工具书式术语解释"的核心特征
       - **典型反例 / 诊断信号**：直接讲"X 实验失败的原因" / 直接讲"为什么 Y 难做"——读者没有"成功是什么样的"基线，"失败"就是抽象的；用户读完 §3.1 反馈"不懂 X 是什么 / 不懂为什么失败"= 跳过了 happy path。
       - **★ 技术背景小节的物理量符号纪律**：本小节不抛陌生物理量符号——陌生符号必须先用认知钩子（量级 / 类比 / 一句话点明好处）替代，精确符号推到 §3.3+ 子原理再给；如必须用符号，加括号注解（"Γ = 自然线宽 ≈ 5 neV, 详见 §5 公式方程"）。
   - **§3.2 ★ 灵魂锚点**：把整项技术的精髓凝练成**一张表或一句话**（例：SERDS 的"坐标对偶性"——谁不动谁被差分消掉；LTRS 的"双作用合一"——同一束光在焦点处同时实现捕获与拉曼探测）。读者记住这一个就抓住了本质。
   - **§3.3+ 子原理**：按该技术展开（如 §3.3 核心机理 A / §3.4 核心机理 B 等）——可变子节，结构由技术本身决定。
4. **原理示意图**
   - **§4.1+ Mermaid（流程 / 因果图）**：算法步骤 / 数据流 / 因果链。**默认方向 `flowchart TD`（从上至下）**——复杂链路用 LR 会让长节点标签字变小、看不清；只有横向对比视图（A vs B 并列）才用 LR。语法踩坑清单见 §三·3.5
   - **§4.2+ SVG**：频谱位置、χ⁽³⁾ 谱形、信号路径、光路几何，等几何 / 谱图类内容。**完整规范（命名 / 字体 / 锚点 / 颜色 / 生成脚本）见 §三·3.3**
   - **★ 禁止用 ASCII art（```代码块围栏的字符图）替代几何 / 谱图**：等宽字体宽度不一导致错位、emoji 与中文混排错位、不可缩放、无颜色编码
5. **基本公式方程**
   - 每个关键公式后尽量给 **带入数值的算例（worked example）**（如 "λ=785 nm, Δλ=0.5 nm → Δν̃≈8.1 cm⁻¹"）
   - ★ **适用区间/有效性警示**：注明公式成立的前提与边界（如瑞利区 vs Mie 区、Kasha 规则例外）。**不给未经数学/物理验证的式子，边界必须写清**
   - 公式四要素（公式本体 / 符号说明 / 数值例子 / 适用条件）的统一写法见 §三·3.1。
6. 技术变体 / 发展（可选）：该技术存在 ≥1 个主流变体时必填；否则省略。同样配图 + 公式 + 对比表。
7. 数据处理算法（可选）：涉及机器学习 / 统计建模 / 化学计量学 / 信号处理等数据驱动方法时做，算法 / 原理 / 优势 / 适用场景 四列表。
8. **典型示例应用**
   - 来源于论文的示例应用，需在 **§10 参考文献** 处提供参考文献，并在此处引用
   - 需列出示例应用涉及到本技术的仪器配置表、样品集、处理策略、关键发现
   - ★ **明确标注"用到本技术/变体的具体位置"**（便于核对"本技术是否真在论文中被使用"）
9. **注意事项与工程要点**：
   - 缺点 / 弱项 / 不足 / 局限束缚，分列
   - ⚠️ **踩坑框** + 表格（"要点 / 原因"两列）
10. **参考文献**：每篇文献必须包含作者 / 标题 / 期刊 / 年份 / 卷期页 / DOI；DOI 必须核实，未核实的一律标注 `（待核实）`；分类排列：经典（按时间正序）→ 综述 → 应用论文；若用户提供了基础文档材料，则基础文档材料排应用类第一条。**完整规范见 §三·3.2。**
11. **依赖地图**：以一张四列表（依赖强度 / 技术 / 在 A 中的角色 / 是否需要独立简报）声明本简报与外部基础技术的关系。格式与判定准则见 §三·3.4。

> 💡 **MD 骨架模板（template.md）**：本 skill 同目录下的 [`template.md`](template.md) 与上述 11 章节列表 **1-to-1 对齐**，以 `{占位符}` 形式给出可直接套用的空白简报结构。
> 
> **推荐工作流**：
> 1. 复制 `template.md` → 重命名为 `技术简报.md`
> 2. 按 §一 顺序，逐章替换 `{占位符}` 内容

## 二、交互演示（交互式 HTML）规格

### 触发条件

当技术的核心原理涉及**"参数变化 → 光谱/信号形态变化"**等因果关系时（例如：波长差影响差分峰幅度、光阱结构影响探测区域），且能用几个滑块让核心物理量变化趋势可视化时，**必须**生成一份独立 HTML 交互演示。

**满足以下任一即必须生成**：① 至少 1 个滑块能改变谱形/几何/信号；② 演示能体现 §3.2 灵魂锚点的对比视图（"正确 vs 错误"、"A vs B" 等）；③ 演示能让读者"看到某个 trade-off 怎么动"；④ 核心机理涉及多参数协同。

**满足以下任一即可省略**：① 技术核心无"参数 → 信号形态"因果链；② 是 SOP 或制备方法；③ 核心是单点公式或常数表；④ 用 Mermaid + SVG 已能完整传达全部原理。

### 技术要求

- **单文件自包含**：HTML + JS + `<canvas>`，**离线双击可开**
- **技术栈**：JS/TS 优先，ESM 风格；若涉及张量 / ML 再考虑 tfjs（浏览器端）
- **中文页面**：标题、轴标签、图例全部中文
- **高 DPI 适配**：见下方「高 DPI Canvas 适配」节
- **参数滑块实时重算重绘**：
  - 简单交互（默认）：`input` 事件 → 重新 `computeData()` → `drawAll()`，vanilla JS 即可，**零依赖**
  - 较复杂交互响应（多 view、组件化、跨滑块共享状态）：**VUE3 渐进式引入**——以 CDN 单文件形式挂载，**不要**走 `npm + Vite/webpack`：
    ```html
    <script src="https://unpkg.com/vue@3/dist/vue.global.js"></script>
    ```
    ⚠️ CDN 引入意味着**首次加载需联网**（`vue.global.js` 浏览器缓存后即可离线）；因此务必**在 HTML 中显式声明 "需要联网"**，让用户知道。
- **围绕"核心洞察"设计对比视图**：配色与内容含义匹配（正确用暖绿/错用暖红等），一眼看懂
- **中文注释**：物理参数、函数、算法流程均中文注释
- **物理要忠实**：演示里的公式实现要和 MD 里的公式一致，不能为了好看而失真
- 自绘坐标轴/网格/图例/零线；峰型用洛伦兹（拉曼峰）/高斯（荧光/带）等符合物理的函数
- 若演示中也用到同样的几何/谱图示意（**非交互态**），直接 **`assets/` 单一来源复用** SVG（详见 §三·3.3 第 4 条），避免在演示里用 canvas 重画一遍静态图。
- **交付前自测**（硬性要求）：演示 HTML 完成后，**必须**在本地浏览器打开、逐个交互 ≥ 1 次、运行 ≥ 10 秒，目视无 bug 后才能提交。常见 bug 清单：
  - 选参数时画面**累积变大/变小**（高 DPI 像素累积陷阱）
  - 滑块拖动时画面**不响应**（事件未绑定 / state 未更新）
  - 循环动画（CV 扫描）**越来越快**（`requestAnimationFrame` 累积未取消）
  - 切换下拉时**canvas 残留**（清屏时未用 dpr 重置坐标系）

> 💡 **环境适配提示**：许多 AI Agent 的浏览器工具不允许通过本地文件路径打开 HTML——若你的工具支持直接打开本地 HTML（`file://` 或本地路径），跳过本提示；若不允许，可用临时 HTTP server 调试。可参考下表按宿主常用工具选其一：
> 
> | 工具 | 命令 |
> |:---|:---|
> | **Node** | `npx http-server <Tech-Name 目录> -p 8765 --silent` |
> | **Bun** | `bun serve <Tech-Name 目录>` |
> | **Python** | `python -m http.server 8765`（需在 `<Tech-Name 目录>` 下执行） |

### 高 DPI Canvas 适配

DPR 适配的完整 `initCanvas` + 重绘模板已封装在 [`template.html`](template.html)（**直接抄，不要重写**）。

关键原则：
- **CSS 控制显示尺寸**（`.demo-canvas-wrap` 的 `aspect-ratio`），**JS 只设物理像素**（`canvas.width = cssW × dpr`）
- **不要在重绘函数里赋值 `canvas.width/height/style.width/style.height`** → 物理像素累积爆炸
- **`ctx.setTransform(dpr, 0, 0, dpr, 0, 0)` 每次重绘前重置** → 避免累积缩放
- **ImageData 索引用整数**（`imgData.width`），不要用浮点 `plotW` → 否则小数索引被 `Math.floor` 落到错误像素的 A 通道，热图 / 位图大面积失色

### Canvas 散点可视化反模式（必读）

演示中大量场景需要画散点（标定曲线、重复测量、PCA 投影等）。**绝对避免的反模式**：

#### ❌ 反模式：`Math.random()` 给散点位置生成随机偏移

```js
// 每次重绘 Math.random() 重新随机化 → 用户拖噪声滑块时散点整体漂移，看不到宽度变化
const x = baseX + (Math.random() - 0.5) * 2;
```

#### ✅ 修复：Mulberry32 seeded PRNG + 模块级预生成 SCATTER_OFFSETS

```js
// Mulberry32 PRNG（固定 seed，可重现）
function mulberry32(seed) {
  return function () {
    seed = (seed + 0x6D2B79F5) | 0;
    let t = seed;
    t = Math.imul(t ^ (t >>> 15), t | 1);
    t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  };
}
// 页面加载时一次性预生成固定偏移（不在重绘里调用 Math.random）
const SCATTER_OFFSETS = Array.from({length: N},
  () => (mulberry32(SEED)() - 0.5) * range);
```

**重绘原则**：位置用 `SCATTER_OFFSETS[i]`（稳定），噪声幅度用确定性方向 × 当前 `noiseSigma`（来自滑块）。

**验证方法**：调噪声滑块时，散点的 **X 轴位置不变**，只 **Y 轴散布宽度变化**。两者都变 = `SCATTER_OFFSETS` 没生效或被覆盖。

#### 配套可视化（强烈推荐）

为让"噪声影响"一眼可见，加 **±σ 误差带**（半透明矩形 + 虚线边界）作为辅助层。

#### 反模式红线（看到必须改）

##### 反模式 A：物理公式简化陷阱（分子分母共用项被约掉）

**症状**：用户调某个滑块，但对应物理量在演示里**不响应**——99% 是 bug 不是简化。

**典型案例**（共振拉曼演示面板 C "A/B-term 占比"）：
```js
// ❌ 错：A 和 B 共用 D = Δν² + Γ²，Δν 在占比里被约掉
const A = 1 / Math.pow(dn*dn + gamma*gamma, 2);  // A ∝ 1/D²
const B = 1 / (dn*dn + gamma*gamma);             // B ∝ 1/D
const aFrac = A / (A + B);                       // = 1/(1+D)，只依赖 D
```

**诊断 4 步**：
1. 用户报告"X 滑块不响应"
2. 展开占比公式看变量：哪些参数在化简后消失
3. 数学验证：把 A/(A+B) 化简，看哪些参数被约掉
4. 跑 `node -e` 数值模拟确认（**不要只靠代码 Review**）

**修复原则**：让分子分母的**频率依赖形状真正不同**（不能共用同一个 D）。例如 A ∝ ΔQ²/D（一阶 Lorentzian），B ∝ Δν²/D²（二阶 Lorentzian × Δν² 加权）。

##### 反模式 B：niceTicks 浮点尾巴（"2.0000000003"）

**症状**：坐标轴标签显示 `2.0000000000000003` 或 `0.6000000000000001` 这类长尾数字。

**根因**：`i * step` 在 `step=0.2` 时浮点累积（`3*0.2 === 0.6` 为 false）。

**修复**（template.html 已实现）：
- 返回 `[{x, label}]` 而不是裸 `number[]`
- `label = x.toFixed(decimals)` 统一小数位
- 整数计数：`ni * niceNorm * mag`（整数 × 整数 × 10ⁿ → 精确）

**教训**：任何要显示给用户的数值都过 `toFixed(decimals)`，不要相信原生浮点的 `.toString()`。

##### 反模式 C：「⚙」标记的备忘混入交付内容

**症状**：sidebar 里写"💡 桌面端：滑块固定在左侧...📱 移动端：滑块在顶部..."——用户看见 UI 自然懂，不需要解释。

**判断标准**：> 这段文字**去掉后用户会不会困惑**？会困惑 → 保留（但放交付内容里）；不会困惑 → 是「⚙」标记的备忘。

**正确做法**（二选一）：
- 直接删除（最干净）
- 保留为 HTML 注释：`<!-- ⚙ 备忘：桌面端 sticky / 移动端切 static -->`

##### 反模式 D：数据超出坐标轴未 clip

**症状**：业务数据的 X/Y 范围超出坐标轴显示范围时，数据线画到坐标轴外（如 Y 轴左侧、X 轴下方），与刻度文字 / 轴标签重叠。

**根因**：drawAxes 不自动 clip 数据绘制——业务层用 `ctx.lineTo(xToPx(x), yToPx(y))` 画线时，超出坐标轴范围的点会被画到 plotArea 外。

**修复**（template.html 已实现）：
- 方式 1：`drawLine(ctx, xData, yData, xToPx, yToPx, opts, plotArea)` — 传入 `plotArea` 参数（drawAxes 返回值），drawLine 自动 clip
- 方式 2：`withClip(ctx, plotArea, () => { /* 手动画线 */ })` — 用 withClip 包裹自定义绘制代码

**判断标准**：数据范围是否可能超出坐标轴显示范围？是 → 必须用 clip。常见场景：
- 绝对波长 / 波数网格的数据范围 > 显示窗口（如 SERDS 中 wlAbs 788-1082 nm，X 轴只显示 810-1020 nm）
- 对数坐标轴的数据跨越多个数量级
- 归一化数据的尾部超出显示范围

> 违反时 agent 应自我反思——**用户觉得不需要 = 我大概率在过度工程**。

| 场景 | ✅ 正确动作 | ❌ 错误动作 |

### 3.2 参考文献规范

| 步骤 | 必须做的 |
|:---|:---|
| 1. 检索 | 用库自带的检索方式查找（**中文检索与英文检索可能使用不同机制**，如 FTS 与 LIKE、unicode 分词差异等，按宿主工具实际能力选用） |

| 4. 未命中 | 才走外部抓取（§3.2.2 决策树） |

> 📌 按顶部「硬边界」原则：上表是跨库通用的"先查 → 取全文 → 讲解"骨架。具体命令 / API 按宿主实际工具选用。

- 用户提供的应用论文排在应用类第一条
- **标注格式统一**：本节用 `（待核实）`；与第四章 §四 "诚实性红线" 第 2 条同名标注格式完全一致，便于跨章节检索与维护。

#### DOI / 全文获取决策树（父级词汇）

具体用什么抓取工具看宿主环境，可参考下面的决策树：

```
DOI / URL 给定
  │
  ├─ 步骤 1：用宿主可用的网页抓取类工具一次尝试（如 fetch 类 / web_fetch 类 / search 类工具等，具体可用工具按宿主环境而定）

  │
  ├─ 步骤 2：换另一种抓取类工具（如 fetch 类失败换浏览器自动化类工具）
  │   ├─ 成功 → 走 §三·3.2 后续规范引用即可
  │   └─ 失败 → 进入步骤 3
  │
  ├─ 步骤 3：尝试备用入口
  │   ├─ 作者主页 / 预印本平台 / 开放获取镜像 / 机构知识库
  │   ├─ 成功 → 走 §三·3.2 后续规范引用即可
  │   └─ 失败 → 进入步骤 4
```

- **不要在第一步失败就放弃**——按决策树至少尝试到步骤 3

- **抓到的内容是 JSON 包装（多模态抓取类工具常见格式）** → 走 `clean.ts --from-mcp` 类清洗工具后人工 review（具体清洗工具名按宿主实际可用工具选用）

### 3.3 SVG 文件管理规范

1. **存放位置**：所有 SVG 一律放 `assets/`，文件名用**纯 kebab-case**（web 习惯）命名（例：`trap-geometries.svg`），便于 git 跟踪与跨平台引用。
2. **用标准图片语法引用**（其它形式**一律禁用**）：
   ```markdown
   ![光路示意图（含受力箭头）](assets/optical-path.svg)
   ```
   - **禁用** `[name](path.svg)` 文字 / 表格链接形式——markdown 渲染器只把它当裸链接、看不到图
   - **禁用** `<svg>...</svg>` 块内嵌到 md——部分渲染器对 inline SVG 支持不一致

   **合格判定**：在任意常用渲染器（VSCode preview / GitHub / Typora / Obsidian）中打开 MD 能直接看到图。**看不到 = 违规**。
3. **跨平台一致渲染的写法约束**：
   - 避免 `xlink:href` 外链；
   - 字体用 `font-family="system-ui, -apple-system, 'Segoe UI', 'PingFang SC', sans-serif"` 兜底，**不用等宽字体强行对齐**；
   - 颜色用显式十六进制或 `currentColor`，不依赖外部 CSS class；
   - 文字标注用绝对像素坐标 + `text-anchor="middle/end"`，**不靠字符宽度对齐**。
4. **脚本存档 & 生成方式**：
   - 脚本存档命名：`scripts/generate-xx.{html,mjs,ts}`（**首选 `.html`**，理由见目录规范 callout）
   - **几何 / 示意 SVG**：手写即可（参考 LTRS 的 `assets/optical-path.svg`）
   - **数据驱动 SVG**：D3.js（天然输出 SVG DOM，零配置）或 echarts（需设置输出 SVG 而非 canvas）
   - **避免 Python matplotlib / Origin / gnuplot**：浏览器 / 手机 / 平板 / GitHub preview 上无法直接复现
5. **文件命名示意**（建议风格）：
   - `optical-path.svg` 光路 / 几何示意
   - `trap-geometries.svg` 对比 point / donut / line 等几何形态
   - `spectrum-soc.svg` 数据谱图（与具体数据强相关）
   - `dual-wavelength-spectrum.svg` 双波长光谱示意
   - `potential-well.svg` 势阱 / 势能面
   - `force-balance.svg` 受力分析
6. **viewBox 留白 + 弹性显示**：
   - **症状**：SVG 文字标注被 viewBox 边界裁切（如"简正坐标"四个字只显示一半）
   - **根因**：viewBox 宽度不够 + 文字位置（`x=` / `text-anchor=start`）超出右边界
   - **修复模板**：
     ```xml
     <svg xmlns="http://www.w3.org/2000/svg"
          viewBox="0 0 W_PADDED H_PADDED"
          preserveAspectRatio="xMidYMid meet"
          style="max-width: W_PADDEDpx; width: 100%; height: auto; display: block; margin: 0 auto;">
     ```
   - **设计原则**：
     - 先定画布内最大元素范围（文字最右端 x 坐标 + 估宽：汉字 ≈ `font-size × 1.0`，英文 ≈ `font-size × 0.55`）
     - 四周加 padding（通常 20-50px）
     - **不要写 `width=` / `height=` 属性** → 用 style 的 `max-width`
     - **保留 `viewBox` + `preserveAspectRatio`** 用于缩放控制
   - **三件套协同**（缺一失效）：
     | 属性 | 作用 |
     |:---|:---|
     | `max-width: <px>` | 限制最大尺寸，避免大屏拉爆 |
     | `width: 100%` | 容器内自动撑开 |
     | `height: auto` | 按 viewBox 比例自适应高度 |
   - **验证踩坑**：默认 Node http server mime 不带 `.svg` → 浏览器显示 broken/文本。**调试 SVG 用 `Read` 工具读 XML 文本对照 attribute 即可**，需要看视觉时用浏览器工具直接打开 SVG 文件 URL——**不要为 mime 自建 server**（违反 §二「Agent 行为约束」）

### 3.4 技术依赖处理规则

#### 核心原则

**每个技术简报只覆盖其「主体技术」，不展开「基础依赖技术」的完整介绍。**

避免教学法漂移（pedagogical drift）：写 EC-SERS 时大段讲拉曼原理，写拉曼时又大段讲分子振动，链式蔓延会让单篇简报膨胀且失焦。

#### 定义

| 角色 | 含义 | 示例（主体=EC-SERS） |
|---|---|---|
| **主体技术 A** | 当前简报直接讨论的技术 | EC-SERS 联用 |
| **基础依赖 B** | A 的实现/理解所依赖的更底层技术 | 拉曼光谱、电化学基础 |
| **弱依赖 C** | 仅引用概念、不需读者深入理解 | LSPR、表面等离激元 |

#### 依赖强度判据（看到就判定）

判别某项依赖属于哪一档，**满足任一即按该档处理**：

| 强度 | 判据（满足任一即归此类） |
|:---|:---|
| **🔴 强依赖** | ① A 的 §3 原理叙述（不含灵魂锚点）必须靠 B 才能讲清楚<br>② A 的公式里有 B 的物理量作为自变量（且不是直接引用）<br>③ A 的实验结果里反复出现 B 的标志性现象 / 术语<br>④ A 的 §3.2 灵魂锚点的关键推论 / 数据来源必须靠 B 才能成立 |
| **🟡 弱依赖** | ① 仅在 A 的某一变体或某一章节涉及 B<br>② A 引用 B 作为补充解释，但不展开 |
| **⚪ 仅提及** | ① A 仅引用 B 作为工具 / 实验条件 / 标准件<br>② B 的存在对理解 A 没有影响 |

> **视角差异**：判据 ① 与 ④ 在多数场景下重合（前者描述 §3 原理叙述，后者锁定 §3.2 灵魂锚点）；视为任一即判定 🔴 强依赖即可。

#### 写作约束

**1. B 在 A 简报中的篇幅上限**

- **总篇幅 ≤ 10%** 或 **≤ 1 段**（取较严）
- 仅写"A 在什么环节用到了 B 的什么能力"，**不展开 B 的原理/历史/完整介绍**
- 给出关键术语 + 一句话定位 + 推荐查阅入口（教材章节/经典综述 DOI），让读者按需跳转

**2. 强依赖标注（简报末尾必填，对应 §11 依赖地图）**

完成主体内容后，在简报末尾添加「依赖地图」小节：

```
## 依赖地图（Dependency Map）

| 依赖强度 | 技术 | 在 A 中的角色 | 是否需要独立简报 |
|---|---|---|---|
| 🔴 强依赖 | 拉曼光谱基础 | 信号产生机理 | ⚠️ 待询问 |
| 🟡 弱依赖 | LSPR | 增强机理引用 | 否 |
| ⚪ 仅提及 | 电化学工作站 | 工具引用 | 否 |
```

**3. 询问而非默认展开**

简报交付时用以下句式询问用户（不自己决定）：

> 本简报在 **[环节]** 处强依赖 **[技术 B]**，但未展开其基础原理。
> 是否需要我单独撰写一份 B 的技术简报？
> （如需，请确认侧重方向：基础原理 / 工程实践 / 文献综述）

**4. 链式触发保护**

- 用户确认写 B 的简报后，**B 简报使用同一规则**，避免反向蔓延到 B 的更基础依赖（如拉曼的"分子振动"）。
- 每篇简报只在「依赖地图」中暴露**一层**依赖，深层依赖在 B 简报里再展开。

**5. 反例（避免的写法）**

❌ 在 EC-SERS 简报里花 30% 篇幅介绍拉曼散射的物理原理

❌ 把"基础铺垫"作为简报第一章（除非用户明确要求入门级）

❌ 不询问就主动展开 B 的独立章节（容易造成 A 简报失焦）

**6. 正例（鼓励的写法）**

✅ EC-SERS 简报里写："信号增强依赖于拉曼散射（基础原理见 [依赖地图]，本文不展开）"

✅ 在简报末尾列出强依赖并询问

✅ 用户问"拉曼是什么"时再触发 B 简报的撰写

### 3.5 Mermaid 踩坑清单

Mermaid 解析器对节点标签、边标签、箭头语法有严格的保留字 / 字符约束。下表为常见踩坑与规避方式：

| 现象 | 根因 | 规避 |
|:---|:---|:---|
| 节点标签 `V1["\|v=1⟩..."]` 解析失败 | `\|` 是边标签保留分隔符，引号内仍可能误识别 | Dirac 符号 `\|v⟩` 改用 `⟨v⟩`（U+27E8 / U+27E9）或省略 `\|` |
| `A -.->.` `==>>` `<<<` 等报错 | mermaid 箭头组合有固定白名单 | 严格用 `-->` `-.->` `==>` `<-->` `--x` 等固定写法 |
| `V1 -.->"\|label\|"\| VIRT` 报错 | 引号内已有 `\|`，又用 `\|` 当边标签分隔符（双重嵌套） | 引号 / `\|...\|` 包围两者选一 |
| `A -->\|text<br/>more\| B` 报错 | 边标签里 `<br/>` 未 quoted | 必须 `A -->\|"text<br/>more"\| B` |
| 节点 ID 与保留字冲突 | `subgraph` `end` `flowchart` 不能作为节点 ID | 用中文节点 ID（新版 mermaid 支持）或加前缀（`sub_` 等） |
| 子图名带括号 / 引号 | `subgraph "name"` 在某些版本失败 | 用 `subgraph name["..."]` 显式 ID + 标签 |
| **边标签内含 `(`, `[`, `]`, `{`, `}` 等保留字符** | mermaid 解析器把这些字符当成节点形状定义开始符（`(` = PS Parallelogram Start，`[` = SQE Square End 等），抛 `Expecting 'SQE'... got 'PS'` | 边标签必须用双引号 `\|"..."\|` 包围；或把括号改成全角 `（）【】`；或用 `·,` 代替 |

#### Mermaid 渲染验证 SOP

> ⚠️ **写完 Mermaid 未实测 = 未交付**（硬性要求）——交付物中残留未渲染的 mermaid 块 = 烂尾。

2. **逐项对照踩坑清单**：把上表逐项扫一遍，特别注意 `\|` / 箭头组合 / 节点 ID / `<br/>` / **边标签内含括号**这五类高频坑。
3. **失败定位**：若渲染失败，根据报错定位到具体行号 → 在原 MD 的 Mermaid 块中标注问题字符 → 修改 → 重新预览。
4. **跨平台一致性**：Mermaid 版本不一致会导致同一段代码在不同环境渲染不同。建议在简报末尾注明使用的 Mermaid 版本范围（如"v10+ 推荐"）。

## 四、诚实性红线（不可破）

### 4.1 核心四条

- **公式、理论，均必须得有出处、依据、或参考文献**，否则不写
- **不确定就标注**：DOI / 路径 / 包名 / API 等"硬事实"未核实时标 `（待核实）`，不靠记忆脑补
- **数学验证底线**：任何拟合公式 / 数学模型都**必须有数学或物理来源、或经过验证**。找不到来源的拟合不写；"应该差不多"不构成理由。这条不可破例（独立 skill 不绑死具体学科——任何技术领域的简报都适用）
- **路径/文件名先用工具核实**再下笔（Read/Bash 等），不靠记忆脑补

### 4.2 数值可视化前的强制验证

任何带具体数字的插图（SVG 标注 / HTML 演示面板的数值结论 / Mermaid 节点标签 / 公式算例），**必须先用某种计算工具验证一次**，并在简报末尾记录验证命令 + 输出结果。

**典型触发场景**：

- SVG 上标"Δν̃=3 cm⁻¹ → cos_sim = 0.57" → 必须先验证 0.57 不是脑补
- 公式算例"代入 a=1.0, b=2.0 → 5.83" → 必须先验算
- HTML 演示面板"SNR=10 dB → cos≈0.7" → 必须先验证
- Mermaid 节点标签"准确率 96.5%" → 必须先验证

**验证工具示例**：
- ✅ **Node 一行**（推荐首选）：
  ```bash
  node -e "console.log(Math.cos(30*Math.PI/180))"   # → 0.8660254037844387
  ```
  适合：JS 语系 `Math` / `Number` 标准库覆盖的场景（三角函数、对数、随机数、复数、BigInt）。
- ✅ **Bun 一行**（已装 Bun 时）：
  ```bash
  bun -e "console.log(Math.PI * 100)"
  ```
  同 Node，Windows 启动略快；零额外依赖。
- ⚪ **Python 一行**（**仅在 JS 生态薄弱时使用**，如 numpy / scipy / matplotlib 这类重型科学计算）：
  ```bash
  python -c "import math; print(math.cos(math.radians(30)))"
  ```
  适合：多维矩阵、信号处理、统计等场景。**注意**：Python 一行需要 `import math`，比 Node / Bun 的 `Math.cos(...)` 多一步；因此**简单算式场景倾向 Node / Bun**。

### 4.3 通用触发式自查清单（看到就触发）

下表是**通用学术诚信规则**，撰写涉及下列内容的章节时，必须自查：

| 触发信号 | 必须做的事 |
|:---|:---|
| 看到 **p < 0.05** 或"显著差异" | 追问 n 是多少？效应量呢？样本是否独立？ |
| 看到 **R² > 0.99** 或"完美拟合" | 反问残差图过没过？过拟合风险评估了吗？ |
| 涉及**方法学验证 / 性能确认**类任务 | 按通用框架走：专属性 / 线性 / LOD/LOQ / 准确度 / 精密度 / 鲁棒性（顺序一个不能少） |
| **LOD/LOQ 数值** | 用 S/N = 3 / 10 等明确标准方法计算，**不能目测外推** |
| **数据与文献不符** | 深挖机理（这是研究机会，不是误差），不要硬调和 |
| **增强因子 / 灵敏度 / 检出限类比值** | 公式完整写出，给出全部来源数据，不省略分母 |
| **DFT / 仿真类数值** | 注明 functional / basis set / 收敛阈值 |
| **体外释放度 / 动力学曲线** | 注明介质 / 温度 / 转速 / 取样时间点等关键实验条件 |

> 📌 按顶部「硬边界」原则：上表是跨学科通用的"看到 X → 必须做 Y"格式。具体应用时（如物理化学、生物、材料、半导体等），按对应领域的对应规则展开。

### 4.4 找不到来源/全文/数据争议时的处置

| 情况 | 处置 |
|:---|:---|
| DOI 验证失败（抓取工具全部失败） | 标注 `（待核实）`；在简报末尾说明尝试过的入口 |
| 找不到全文（仅有摘要） | 标注 `（仅摘要）`；引用时**仅引用摘要明确陈述的结论**，不脑补推论 |
| 数据来源有争议 | 列出多方来源 + 各自表述 + 标注差异，不擅自调和 |
| 算例的数值算出来跟文献不符 | **不要硬调和**——按 §4.3 第 5 行"数据与文献不符 → 深挖机理"处理，在简报末尾记录 |
